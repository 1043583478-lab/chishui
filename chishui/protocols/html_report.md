# HTML 报告生成规范

> 定位: 拆局协议的交付层

## 何时生成 HTML 报告

- 用户明确要求 HTML 报告
- 用户说"帮我保存"、"帮我整理成文档"、"帮我出报告"
- 问题复杂度高（多角色、多阶段、需要可视化）
- 用户需要把分析结果发给第三方

## 不生成 HTML 的情况

- 用户只需要快速判断
- 问题还很模糊、澄清未完成
- 用户明确只要文字分析

---

## HTML 报告标准模块

### 必选模块

#### 1. 封面与总判断
- 问题场景一句话描述
- 当前阶段判定（防御/相持/反攻 or 耕种模式）
- 服务对象（这份报告写给谁看的）
- 当前推荐路线（一句话）

#### 2. 核心判断
- 主要矛盾是什么
- 为什么这个是主要的
- 用户原始问题和重构后问题的对照

#### 3. 推荐路线
- 主路线：方向 + 顺序 + 关键支点
- 成立前提（什么条件下这条路是对的）
- 近期动作（3-5 个具体行动 + 观察点）

### 可选模块（按需要添加）

#### 4. 关系图
- 谁影响谁、谁卡谁、谁依赖谁
- 用 Mermaid 或 ASCII 图
- 标注关键控制点和断裂点

#### 5. 时间线
- 关键事件的时间序列
- 标注转折点和触发事件

#### 6. 路线比较表
- 2-3 条可选路线
- 每条：成立前提、所需资源、核心风险、退出条件

#### 7. 控制点分布表
| 控制点 | 名义归属 | 现实掌握方 | 差距原因 | 能否改变 |
|--------|----------|------------|----------|----------|
| ...    | ...      | ...        | ...      | ...      |

#### 8. 证据链
- 每个关键判断列出支撑的事实/数据/事件
- "为什么这样判断"的可追溯依据

#### 9. 人物与关系清单
- 各方诉求、态度、依赖
- 合作边界和红线

#### 10. 方法出处
- 报告中使用的分析方法是来自《毛选》哪篇文章
- 增加可信度和可追溯性

## 设计规范

### 视觉风格
- 红白主色调（参考毛选经典配色：深红 `#8f140c`、暖金 `#c99745`）
- 模块之间用明显分隔
- 每个模块一个独立卡片/区块

### 文件格式
- 单文件 HTML，所有 CSS/JS 内联
- 可直接保存、离线查看、微信转发
- 不依赖任何外部 CDN 资源

### 代码规范
- UTF-8 编码
- 语义化 HTML5 标签
- 响应式布局（移动端可读）
- CSS 打印样式（可打印为 PDF）

### 视觉风格
红底金字配色，模块化卡片布局，语义化 HTML5。详见上方模块规格。

## 生成流程

```
用户确认要 HTML 报告
        │
        ▼
┌──────────────────────┐
│ 检查关键结构是否完整   │
│ (目标/事件/人物/约束)  │
└──────┬───────────────┘
       │
       ├── 不完整 → 先补齐结构，明确告知用户还需要什么
       │
       ▼ 完整
┌──────────────────────┐
│ 走拆局协议完成分析    │
│ (矛盾/阶段/力量/风险) │
└──────┬───────────────┘
       │
       ▼
┌──────────────────────┐
│ 选择模块组合          │
│ (根据问题类型决定     │
│  哪些可选模块要添加)  │
└──────┬───────────────┘
       │
       ▼
┌──────────────────────┐
│ 生成单文件 HTML       │
│ 所有 CSS/JS 内联      │
└──────┬───────────────┘
       │
       ▼
┌──────────────────────┐
│ 交付给用户            │
│ (可保存/转发/打印)    │
└──────────────────────┘
```
